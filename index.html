Supply Chain · Statistical Modelling · Dashboard

Nassau Candy Distributor — Shipping Route Analytics

Full-cycle supply chain analytics project — analysed 8,549 shipments across 5 factories, 4 regions, and 16 routes. Built a composite efficiency score combining lead time and delivery variability, with a custom Z vs t-distribution confidence framework applied across all route aggregations.

8,549 shipments 16 routes ranked 5 factories · 4 regions
PythonPandasSciPy PlotlyStreamlitStatistical Modelling
Live app GitHub Mar - April 2026
Problem

A distributor shipping confectionery products from 5 factories across the US needed to understand which factory→region and factory→state routes were genuinely efficient — and which were bottlenecks. The challenge was doing this rigorously when some routes had 1,500+ shipments and others had fewer than 20.

Challenges

The sample size problem. Applying the same confidence interval logic to a route with 19 shipments as one with 1,524 produces misleading rankings. Every threshold in the analysis also needed to be data-driven — set from the actual percentile distribution of the data, not hardcoded arbitrarily.

Process

Data cleaning (corrupted dates, cart-update duplicates, misclassifications) → Three-way merge (orders + products + factories) → Percentile analysis → Custom Z vs t-distribution framework → Efficiency score engineering (speed + consistency) → Route ranking across 6 dimensions → Streamlit dashboard

Key Insight

Efficiency is not the same as speed. A composite score combining normalised lead time and delivery variability surfaces routes that are both fast and predictable — the ones a business can actually rely on. Using the upper-bound confidence interval throughout ensures conservative, operationally sound planning.

Value

8,549 shipments analysed · 16 factory→region routes ranked · Statistically rigorous confidence framework with automatic Z vs t selection · Real-world data quality issues diagnosed and resolved · Fully interactive Streamlit dashboard with live filters.

Forecasting · ML · Deployed

Real Estate Price Forecasting & Recommendation Engine

Scraped 99acres.com, engineered 15+ features, trained an XGBoost model, and deployed a live Streamlit app — with a recommendation engine that surfaces contextually similar properties alongside each prediction. A single-output model turned into a decision-support tool.

~50% MAE reduction R² ≈ 0.90 Live deployed
PythonXGBoostStreamlit Scikit-learnWeb ScrapingCosine Similarity
Live App GitHub Nov – Dec 2025
Problem

Property pricing on platforms like 99acres is largely opaque — listings exist but there's no easy way to know whether a price is fair, overpriced, or what comparable alternatives look like. The goal was to build a tool that could take a property's attributes and return both a predicted price and a ranked list of similar properties.

Challenges

The real challenge wasn't the model — it was the data. Scraped real estate data is messy in ways that aren't obvious: properties listed in wildly different formats, prices that include or exclude taxes inconsistently, room counts recorded differently across listing types. Getting the features clean enough to be meaningful took most of the project time.

Process

Web scraping → Raw data cleaning → Feature engineering (15+ features) → XGBoost model training → Evaluation → Recommendation layer (cosine similarity) → Live Streamlit deployment

Key Insight

Room segmentation was the single highest-impact feature engineering decision. The recommendation layer changed the nature of the tool: instead of a single prediction, the output became a comparison — here's your estimate, and here are five real listings that are similar — which is far more useful for an actual decision.

Value

~50% MAE reduction over baseline · R² ≈ 0.90 · Live deployed. Turned a regression model into a decision-support tool — a single number is informational, but a ranked list of alternatives is actionable.

MLOps · Production · Architecture

End-to-End Demand Forecasting System

A production-grade forecasting system built across three decoupled, containerised services — model execution, FastAPI inference, and Streamlit visualisation — orchestrated with Docker Compose and deployable in one command. Bulk ingestion via PostgreSQL COPY cut load time by 10×.

36% MAE reduction 10× faster ingestion 3 decoupled services
PythonXGBoostPostgreSQL FastAPIDockerStreamlitNeon
Live App GitHub Jul – Aug 2025
Problem

Most forecasting projects end at a Jupyter notebook — a trained model that lives on someone's laptop and gets manually re-run. The goal here was to build a system — something production-ready, modular, and deployable by anyone with a single command.

Challenges

Designing the service boundaries correctly. The temptation is to couple the model and the API — it's simpler to build. But coupling means you can't update the model without touching the serving layer. Getting the single-responsibility split right required thinking about how the system would change over time, not just how it would work on day one.

Process

Data ingestion (PostgreSQL COPY) → Feature engineering → XGBoost model training → FastAPI inference layer → Streamlit visualisation → Docker Compose orchestration → One-command deploy

Key Insight

PostgreSQL COPY vs row-wise inserts is an underappreciated performance decision — the 10× load time reduction came entirely from switching ingestion method, with no model changes needed. The microservices pattern, even at small scale, forces cleaner thinking about what each component owns.

Value

36% MAE reduction (MAE: 110 → 70) · 10× faster ingestion · Fully containerised, production-deployable in one command. Architecture reflects real engineering standards, not notebook-to-production shortcuts.

MSc Dissertation · Time-Series · Research

5G Network Energy Consumption Forecasting

MSc dissertation — modelled energy consumption patterns across 5G network infrastructure using time-series regression and feature engineering. Results confirmed statistically significant at p < 0.0001 across 50+ base stations.

47% MAE reduction p < 0.0001 50+ base stations
PythonTime-SeriesRegression Feature EngineeringGroupKFoldStatistical Testing
GitHub Feb – Jul 2025
Problem

5G network infrastructure consumes significantly more energy than previous generations. Operators need to predict consumption patterns across base stations — both to manage costs and to identify efficiency opportunities. The dissertation asked: can we accurately forecast energy consumption at the base station level, with statistically rigorous results?

Challenges

The data leakage problem. In a dataset with 50+ base stations, naïve cross-validation would train on observations from the same station it tests on — producing optimistic results that don't reflect real-world deployment. Solving this correctly was the difference between publishable results and inflated ones.

Process

Data exploration → Feature engineering (temporal + network patterns) → Baseline modelling → Time-series regression → GroupKFold cross-validation (no site overlap) → Statistical significance testing (p < 0.0001)

Key Insight

Feature engineering on temporal patterns — time-of-day, day-of-week, network load cycles — contributed more than model complexity. GroupKFold is the correct validation strategy for grouped time-series data, but it's rarely used in practice — most tutorials use standard K-fold which leaks across groups.

Value

47% MAE reduction · p < 0.0001 statistical significance · Clean methodology across 50+ base stations. Dissertation-grade rigour — results are defensible, not just impressive-looking.
